## 1. 问题根因分析

### 1.1 根本原因：Maven 构建 OOM / 超时

**文件**：`blog-backend/Dockerfile`

原 Dockerfile 直接执行 `mvn clean package`，没有分层利用 Docker 缓存。云容器通常只有 **512MB~1GB 内存**，而 Maven 编译 + 依赖下载的内存峰值 **可能超过 2GB**，导致容器被 OOM Killer 终止。

- **下载依赖**：首次需从 Maven Central 拉取 ~50MB 依赖
- **编译 + 打包**：Javac 编译 + Spring Boot repackage，需要大量堆内存
- **无缓存**：`pom.xml` 不变时每次仍重新下载，浪费带宽和时间

### 1.2 连接泄漏导致 HikariCP 连接池耗尽

**文件**：`blog-backend/src/main/java/com/personalblog/config/DataInitializer.java`

```java
// 原代码：getConnection() 后未 close()
ScriptUtils.executeSqlScript(dataSource.getConnection(), schemaResource);
ScriptUtils.executeSqlScript(dataSource.getConnection(), dataResource);
```

`DataSource.getConnection()` 获取的连接**从未调用 `.close()`**，导致连接被泄漏。虽然 `CommandLineRunner` 只运行一次，但在连接池有限的云环境中，这可能阻塞后续数据库操作。

### 1.3 种子数据每次重启重复插入

`data.sql` 使用 `INSERT INTO` 而非幂等语句。容器重启时，`DataInitializer` 会尝试再次插入已存在的数据，导致**主键冲突异常**（虽然在 catch 中被吞掉，但日志不清晰）。

### 1.4 SQL 查询逻辑错误

**文件**：`blog-backend/src/main/java/com/personalblog/mapper/ArticleMapper.java`

```sql
-- 原 SQL（OR 优先级低于 AND，逻辑错误）
WHERE deleted = 0 AND status = 1 
  AND title LIKE '%keyword%' OR content LIKE '%keyword%'
-- 实际等价于：
WHERE (deleted = 0 AND status = 1 AND title LIKE '%keyword%') 
   OR (content LIKE '%keyword%')
-- 正确应为：
WHERE deleted = 0 AND status = 1 
  AND (title LIKE '%keyword%' OR content LIKE '%keyword%')
```

### 1.5 误将 CLI 工具放入 production 依赖

**文件**：`blog-frontend/package.json`

`shadcn` 是 CLI 开发工具，存放在 `dependencies` 中会导致 `npm ci --production` 时安装不必要的包，增大容器镜像体积。

## 3. 详细修改说明

### 3.1 Dockerfile — 添加 Maven 依赖缓存层

**改动**：在 `COPY src` 之前先 `COPY pom.xml` 并执行 `dependency:go-offline`

```dockerfile
# 新增：利用 Docker 层缓存，pom.xml 不变时跳过依赖下载
COPY pom.xml .
RUN mvn dependency:go-offline -q

COPY src ./src
RUN mvn clean package -DskipTests -q
```

**原理**：
- Docker 的镜像层缓存基于**文件变化**判定
- `COPY pom.xml .` 层：只要 `pom.xml` 不变，该层缓存命中，后续 `RUN` 层也命中
- 后续迭代开发修改 `src/` 时，无需重新下载依赖，构建时间从 **~5 分钟降至 ~30 秒**
- 云容器内存压力降低：`dependency:go-offline` 只下载不编译，峰值内存更低

### 3.2 DataInitializer — 修复连接泄漏 + 幂等初始化

**改动 1**：使用 `try-with-resources` 自动归还连接

```java
// 修复前：Connection 泄漏
ScriptUtils.executeSqlScript(dataSource.getConnection(), schemaResource);

// 修复后：try-with-resources 自动 close()
try (Connection conn = dataSource.getConnection()) {
    ScriptUtils.executeSqlScript(conn, schemaResource);
}
```

**改动 2**：执行 `data.sql` 前先检查数据库是否为空

```java
// 新增：article 表有数据时跳过种子数据插入
if (dataResource.exists() && isDatabaseEmpty(conn)) {
    ScriptUtils.executeSqlScript(conn, dataResource);
}

// 辅助方法
private boolean isDatabaseEmpty(Connection conn) {
    try (Statement stmt = conn.createStatement()) {
        var rs = stmt.executeQuery("SELECT COUNT(*) FROM article");
        if (rs.next()) return rs.getInt(1) == 0;
    } catch (Exception ignored) { /* 表不存在 → 首次运行 */ }
    return true;
}
```

**改动 3**：用 SLF4J Logger 替代 `System.out.println`

```java
private static final Logger log = LoggerFactory.getLogger(DataInitializer.class);
// ...
log.info("schema.sql executed successfully");
log.warn("Data initialization skipped: {}", e.getMessage());
```

### 3.3 ArticleMapper — 修复 SQL 优先级

```sql
-- 修复前（OR 无括号）
SELECT * FROM article WHERE deleted = 0 AND status = 1 
    AND title LIKE CONCAT('%', #{keyword}, '%') 
    OR content LIKE CONCAT('%', #{keyword}, '%')

-- 修复后（OR 正确加括号）
SELECT * FROM article WHERE deleted = 0 AND status = 1 
    AND (title LIKE CONCAT('%', #{keyword}, '%') 
         OR content LIKE CONCAT('%', #{keyword}, '%'))
```

### 3.4 package.json — shadcn 移入 devDependencies

```json
// 修复前：在 dependencies 中
"dependencies": {
    "shadcn": "^4.11.0",
    // ...
}

// 修复后：移到 devDependencies
"devDependencies": {
    "shadcn": "^4.11.0",
    // ...
}
```

修复后执行 `npm ci --production` 或生产部署时不再安装 `shadcn`。

## 4. 验证方法

### 4.1 本地验证构建

```bash
# 验证后端 Docker 构建
cd blog-backend
docker build -t starblog-backend:test .

# 验证完整 Compose 启动
cd ..
docker compose up -d --build

# 查看日志
docker compose logs backend
docker compose logs frontend

# 测试 API
curl http://localhost:8081/api/article/list

# 测试前端
curl http://localhost:3002/
```

### 4.2 关键检查点

| 检查项 | 预期结果 |
|--------|---------|
| `docker build` 第二次构建时间 | 显著缩短（缓存命中） |
| 后端容器日志 | 无连接泄漏警告 |
| `GET /api/article/list` | 返回文章列表（状态码 200） |
| 前端页面加载 | 正常显示文章卡片 |
| 容器重启后数据 | 评论/文章不丢失、不重复 |

### 4.3 云容器部署注意事项

- 云容器的 **启动超时时间** 建议设为 **10 分钟以上**（首次构建需下载 Maven 依赖）
- 确保云环境可以访问 `https://repo.maven.apache.org` 和 `https://registry.npmjs.org`
- 如果云平台支持 **预构建镜像**（如阿里云 ACR、腾讯云 TCR），建议先在本地 `mvn package` 后上传 pre-built 镜像

## 5. 部署建议

### 5.1 短期（当前修复）

直接使用本修复后的代码部署，Docker 构建缓存层可大幅降低 OOM 风险。

### 5.2 中期（推荐）

将 Dockerfile 改为**预构建 + 纯运行时镜像**策略：

```bash
# 在 CI/CD 中分两步
# Step 1: 在构建机上编译
mvn clean package -DskipTests

# Step 2: 使用 slim runtime 镜像部署（无需 Maven）
# 修改 Dockerfile 直接 COPY 本地 target/*.jar
```

### 5.3 长期

考虑迁移至 MySQL 替代 H2（正式生产环境），避免 H2 文件锁和兼容性问题：

```yaml
# docker-compose.yml 中启用 MySQL 服务并配置 backend 环境变量
# 取消注释 db service 章节并配置数据库连接即可
```
